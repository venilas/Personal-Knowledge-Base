# Python `logging` — Стандартная библиотека логирования

## Overview / What

`logging` — встроенная библиотека Python для регистрации событий во время работы программы.  
Позволяет записывать отладочные сообщения, предупреждения, ошибки и критические события — как из собственного кода, так и из сторонних модулей.

**Ключевые особенности:**
- Иерархическая система логгеров (root → parent → child)
- Гибкая маршрутизация через Handler'ы (файл, консоль, сеть и т.д.)
- Форматирование вывода через Formatter
- Потокобезопасность из коробки

---

## When to use / When NOT to use

**✅ Использовать:**
- Backend-сервисы (FastAPI, workers, фоновые задачи)
- Long-running процессы
- Distributed systems (трассировка событий)
- Отладка production-инцидентов

**❌ Избыточно:**
- Простые одноразовые скрипты
- Учебные проекты без требований к наблюдаемости
- CLI-утилиты без сложной логики

---

## Основные компоненты

```
Logger → Handler → Formatter
           ↓
        Filter (опционально)
```

| Компонент   | Роль                                                        |
|-------------|-------------------------------------------------------------|
| `Logger`    | Точка входа. Принимает сообщения и передаёт Handler'ам      |
| `Handler`   | Определяет **куда** писать (файл, консоль, email, ...)      |
| `Formatter` | Определяет **как** форматировать сообщение                  |
| `Filter`    | Дополнительная фильтрация записей (опционально)             |

## Уровни логирования

| Уровень    | Значение | Назначение                                     | По умолчанию |
|------------|:--------:|------------------------------------------------|:------------:|
| `DEBUG`    | 10       | Подробная отладочная информация                | ❌           |
| `INFO`     | 20       | Подтверждение штатной работы                   | ❌           |
| `WARNING`  | 30       | Потенциальная проблема, программа работает     | ✅           |
| `ERROR`    | 40       | Ошибка, часть функциональности недоступна      | ✅           |
| `CRITICAL` | 50       | Критическая ошибка, программа может упасть     | ✅           |

> Выводятся все сообщения с уровнем **≥** установленному. Дефолтный уровень root-логгера — `WARNING`.

## Pitfalls

- **`basicConfig` не работает повторно** — настраивает root-логгер только один раз. Повторный вызов игнорируется, если хендлеры уже добавлены. Используй `force=True` для сброса.
- **Не используй root-логгер в библиотеках** — `logging.warning(...)` напрямую загрязняет root. В библиотеках всегда `logging.getLogger(__name__)`.
- **Уровень логгера vs уровень хендлера** — фильтруют независимо. Если логгер на `ERROR`, хендлер с `DEBUG` всё равно не увидит `DEBUG`-сообщения.
- **`print` вместо `logging`** — `print` нельзя фильтровать, форматировать и перенаправлять.

---

## Notes

- Стандартный паттерн для каждого модуля: `logger = logging.getLogger(__name__)`
- Дочерний логгер **пробрасывает** сообщения родителю (`propagate=True` по умолчанию)
- Иерархия строится по имени через точку: `app` → `app.db` → `app.db.query`
